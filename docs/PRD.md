# SceneWeaver 产品需求文档（PRD）

> 版本 v0.1 · 2026-06-05 · 状态：草案
> 配套文档：[剧本 YAML Schema 设计文档](剧本Schema设计文档.md)

---

## 1. 背景与问题

小说作者常希望把作品改编成剧本，但改编是专业活：要把"靠文字写内心"的小说，转成"只能被看见、被听见、被表演"的剧本（删内心独白、外化为动作、重写对白、拆成戏剧化场景）。门槛高、耗时长。

现有"小说转剧本"工具（如海马轻帆）多是**黑盒一键出纯文本初稿**，面向微短剧量产——能转，但**转完难以继续打磨**：无结构、无法定位、无法追溯到原文。

## 2. 产品定位（一句话）

**给作者的、可溯源可打磨的结构化小说改编助手**——把 3 章以上小说自动改编成结构化、可编辑、可追溯原文的剧本（YAML）。

## 3. 目标与非目标

**目标**
- 3 章以上小说文本 → 结构化剧本 YAML，可编辑、可打磨。
- 定义并论证一套剧本 YAML Schema（见 Schema 设计文档）。
- 让"机器生成 + 作者打磨"形成闭环，而非一次性黑盒。

**非目标**（明确不做，控制范围）
- 不做实时多人协作、不做账号/登录体系（本地优先、单用户）。
- 不做配音、分镜、AI 绘图等下游环节。
- 不追求"一键终稿"——定位是**高质量初稿 + 可打磨**。
- 首版不做多语言（中文优先）。

## 4. 目标用户与核心场景

- **用户**：网文/小说作者、改编编剧、影视开发的初筛人员。
- **核心场景**：作者把自己 ≥3 章的小说丢进来 → 快速得到结构化剧本初稿 → 对照原文逐场打磨 → 导出 YAML / 国内排版剧本，作为可继续加工的底稿。

## 5. 竞品与差异化

| 维度 | 海马轻帆（标杆） | SceneWeaver |
|---|---|---|
| 输出 | 纯文本剧本 | **结构化 YAML**（可程序化编辑） |
| 人物一致性 | 黑盒 | **全局人物表 + id + 别名**，跨章一致 |
| 可溯源 | 无 | **每场回指原文段落** |
| 可信度 | 无 | **置信度 + 审阅状态**，低置信标红 |
| 过程 | 一键黑盒 | **双阶段可解释**（先理解后改编） |
| 定位 | 给微短剧厂的量产机 | **给作者的可打磨改编助手** |

## 6. 产品形态与信息架构

**形态**：一个**改编工作台**（不是一键生成器）——生成只是工作台里的一个动作，重心在对照打磨。

**页面（2 路由 + 1 浮层）**
- `/` 首页 = 项目库「我的剧本」（入口即此，无营销落地页）。
- `/project/:id` 工作台（核心单页）。
- ⚙ 设置（弹窗/抽屉）。

**工作台四区**：导入区 / 理解区（人物表）/ 剧本区（主舞台）/ 导出区。
**长内容交互**：以「章」为主导航单元，逐章分页；点剧本场景 → 左侧原文自动滚动并高亮源文段落；每场可折叠成大纲。

## 7. 核心用户流程

```
新建改编 → 导入小说(粘贴/上传, 自动分章分段)
        → 理解：自动切场景 + 抽人物表，用户确认/合并别名
        → 一键生成剧本(双阶段 pipeline)
        → 工作台：左原文/右剧本对照，点场景跳原文，就地改，低置信标红，合并/拆分本场
        → 导出 YAML + 国内排版
        → 自动持久化(IndexedDB)，可回库继续
```

## 8. 功能需求

> 标记：**[MVP]** 必须跑通 · **[加分]** 有时间再上

### 8.1 项目库 [MVP]
- 项目卡片列表（标题 / 进度 / 最后编辑），打开、删除。
- IndexedDB 持久化，刷新不丢失。
- 空状态：一句话价值说明 + 「+新建改编」。

### 8.2 导入 [MVP]
- 粘贴或上传小说文本（≥3 章）。
- 确定性分章分段、段落编号（供源文映射用）。

### 8.3 理解：人物表 + 场景切分 [MVP]
- 抽取人物表（标准名 + 别名 + 小传 + 角色定位），**用户可改名/合并别名**。
- 自动切分场景（按地点/时间切换判断边界）。
- [加分] 原文上可视化拖拽调整场景边界。

### 8.4 生成：双阶段 pipeline [MVP]
- 理解层：逐章扫描 → 维护滚动 story bible（人物 / 情节 / 时间线）。
- 改编层：每场 = 当前场原文(逐字) + story bible(精简) + 人物表 → 输出该场剧本 YAML（删内心戏、外化动作、重写对白）。
- 每场带 `confidence` 自评、`source` 溯源、`adaptation_note`。

### 8.5 剧本工作台 [MVP]
- 按场渲染剧本；点场景 → 原文联动高亮。
- 就地编辑字段（场标 / 动作 / 对白 / 括号提示）。
- 低置信场景标红提示。
- 轻量「合并 / 拆分本场」兜底切错。
- [加分] 逐场「重新生成」（最小化 patch，只改本场）。

### 8.6 导出 [MVP]
- 导出 YAML（数据契约）。
- 渲染为国内排版剧本（场标顺序、△、加粗）。
- [加分] 好莱坞排版 / PDF 导出。

### 8.7 设置 [MVP]
- 配置 LLM 提供方 + API Key（存本地，永不入库）。
- 默认渲染格式（国内 / 好莱坞）。

## 9. 数据契约

剧本数据结构与每个字段的设计理由见 **[剧本 YAML Schema 设计文档](剧本Schema设计文档.md)**。核心：结构化场次 + 全局人物表 + 有序异构 elements + 源文映射 + 审阅元信息；数据只存结构，排版交渲染层。

## 10. 技术方案概述

- **架构**：双阶段 pipeline（理解层 → 改编层）+ 编辑层；长文本靠确定性分块 + 滚动 story bible（摘要只用于"全局上下文"，改编喂原文逐字）。
- **算力分布**：代码管"何时/怎么算"（分章分段、分块调度、别名落库、YAML 序列化与 Schema 校验、规则校验、源文映射、排版渲染）；LLM 管"做什么"（抽人物、切场景、改写对白、外化内心、自评置信度）。
- **栈（待定，倾向）**：单个 Next.js (TS) 全栈，API route 调 LLM，结构化输出后序列化 YAML。
- **模型（待定）**：优先确认七牛是否提供推理 API；否则备 Claude / DeepSeek / Kimi（注意国内模型对冲突/情感题材的 contentFilter）。
- **复用 copychat**：AIMD 限流器、JSON 容错三件套、本地算+LLM判、时态隔离、输出后自检+自动重试。

## 11. 范围：MVP vs 加分项

- **MVP 主线**：项目库 → 导入 → 人物表确认 → 一键生成 → 对照打磨（就地改+标红+合并拆分）→ 导出 YAML + 国内排版，全程 IDB 持久化。
- **加分项**：好莱坞排版、PDF 导出、逐场重新生成、原文场景边界拖拽编辑。

## 12. 验收标准

- **端到端**：一篇 ≥3 章小说 → 生成结构化 YAML，**Schema 校验通过**，每场可溯源到原文段落，可导出国内排版剧本。
- **人物一致性**：同一角色的别名正确归一到同一 id；不同场引用一致。
- **可打磨**：就地编辑生效、IDB 持久化、可重新导出。
- **健壮性**：LLM 返回脏数据不崩溃（容错降级）；超长输入不超窗口（分块）。

## 13. 风险与对策

| 风险 | 对策 |
|---|---|
| 长文超上下文窗口 | 确定性分块 + 滚动 story bible，改编只喂当前场原文 |
| 国内模型 contentFilter 拒答 | 先测拒答率，备 Claude/DeepSeek/Kimi |
| 3 天工期紧 | 先跑通 MVP 核心闭环，加分项最后上；先糙 UI 验证核心 |
| 改编质量参差 | 置信度标红 + 人工打磨兜底（定位本就是"初稿+打磨"） |

| 评分维度 | 权重 | 本作对应 |
|---|---|---|
